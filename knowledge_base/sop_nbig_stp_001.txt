# SOP-NBIG-STP-001: New Business Pre-Issue Checks & Decisioning

## Purpose
Step-by-step checklist and decision rules for New Business pre-issue checks.
Designed to be AI-readable for rule-by-rule reasoning.

## Scope
Applies to QnB, EzSub, and Hardcopy channels. Covers document checks, L400 validations, follow-up code checks, Jet underwriting checks, and decision routing.

## Definitions
- FileNet: Document repository for identity and core documents
- DotSphere: Application / workflow interface
- L400: Policy administration system (source of truth for client/policy fields)
- RCS: Claims system for claim/ICD checks
- MyInfo: Customer consented data source — used to validate consent and compare key data
- BI: Benefit Illustration — premium & benefit illustration document
- PF: Proposal Follow-ups — outstanding requirement codes. Blocks case locking when not resolved
- ANB: Age Next Birthday — pricing age used for premium
- SA: Sum Assured — coverage amount
- NRIC: National Registration Identity Card
- GNS: Government Name Screening — triggered by compliance follow-up codes
- Claim Ind: Claims indicator flag — potential risk signal
- RCS: Risk/claim screening outcome — Pass/Refer based on ICD mapping
- STP: Straight-through processing (no underwriter review)

## Follow-Up Code Reference (Core Codes)
- F45: Premium category — premium mismatch detected (most common FUP code, 53 occurrences in NB.MASTER)
- C09: Compliance category — GNS re-screening required
- CSL: Customer category — clarification required
- AT3: Additional requirement pending
- LIA: Liability / additional info required (51 occurrences)
- PAC: Pending admin completion (44 occurrences)
- UWO: Underwriting outstanding (44 occurrences)
- LOD: Documents outstanding (41 occurrences)
- APS: Attending Physician's Statement required (10 occurrences)
- MEN: Medical examination needed (11 occurrences)
- ECG: ECG test required (6 occurrences)
- B90: Benefit/rider review (17 occurrences)
- See ref_followup_codes.txt for the complete dictionary of 93 follow-up codes.

## Billing Reference Codes (from NB.MASTER)
- BILL_FREQUENCY_CD: 01=Annual, 02=Semi-Annual, 04=Quarterly, 12=Monthly, 00=Single Premium
- BILL_CHANNEL_CD: B=Bank (GIRO/Direct Debit), D=Direct (Credit Card/Cash)

## Channel Gating Rule
IMPORTANT: Each step has channel flags (QnB=Y/N, EzSub=Y/N, Hardcopy=Y/N).
If the channel flag is N for the current channel, the AI MUST SKIP that step entirely.
Do NOT evaluate skipped steps as Pass or Fail — mark them as "Skip".
This prevents false failures when a step does not apply to a channel.

## How to Use
1) Determine the channel from the case data
2) For each step, check if the channel flag = Y. If N, skip it.
3) For applicable steps: read Input Data + Rule IF/THEN, return Pass / Fail / Refer Ops / Refer UW / Skip
4) If an Exception is triggered, record it and recommend next action (do NOT fix data)
5) Aggregate all results using the Decision Mapping to determine final recommendation
6) If any rule/threshold is missing or unclear, flag a Knowledge Gap (do NOT guess)

## AI Usage Notes
- AI must not assume if timestamp is missing
- FileNet documents: in student scope, metadata only (exists/readable)
- L400 fields: read-only — AI evaluates fields, never modifies
- DotSphere: AI recommends decisions, human executes
- PF codes: if not all R → Further Requirements
- ANB/SA: used only in threshold rules
- UW indicators: if Y/M → Refer UW
- RCS/ICD: AI uses result only, not medical judgement
- Workflow: AI triggers workflow recommendation, human confirms

## STEPS & RULES (DETAILED MATRIX)

### Category 1: Open Documents

Step 1A — Open Identity Document on FileNet
- Input: Identity Doc Status
- Source: FileNet
- AI Role: Evaluate
- Rule IF: Document exists → THEN: Pass
- Exception: Missing → create FUP → Decision Impact: Further Requirements
- QnB=NO, EzSub=YES, Hardcopy=YES

Step 1B — Open Benefit Illustration on FileNet
- Input: BI Status
- Source: FileNet
- AI Role: Evaluate
- Rule IF: Exists & readable → THEN: Pass
- Exception: Missing → create FUP → Decision Impact: Further Requirements
- QnB=NO, EzSub=YES, Hardcopy=YES

Step 1C — Open Application Form on DotSphere
- Input: Application Form data + L400 data
- Source: FileNet + L400
- AI Role: Evaluate
- Rule IF: Exists → THEN: Pass
- Exception: Missing → create FUP → Decision Impact: Further Requirements
- QnB=NO, EzSub=YES, Hardcopy=YES

### Category 2: Document Checks

Step 2A — Check MyInfo Consent Form Timestamp
- Input: Consent Timestamp + App Date
- Source: MyInfo, App Form, FileNet
- AI Role: Evaluate
- Rule IF: Timestamp <= 1 year from app date → THEN: Pass
- Exception: Invalid → create FUP → Decision Impact: Further Requirements
- AI Note: AI must not assume if timestamp is missing
- QnB=NO, EzSub=YES, Hardcopy=YES

### Category 3: Client Number Search

- Source: L400

- Exception: 0 or >1 results → Exception → Decision Impact: Refer Ops
- QnB=NO, EzSub=YES, Hardcopy=YES

Step 4A — Search client on L400 using Surname + Given Name

- Source: L400

- Exception: None → Decision Impact: Refer Ops
- QnB=YES, EzSub=YES, Hardcopy=YES

Step 4B — Check Birth Date and Sex in L400 vs Application Form
- Input: DOB + Sex
- Source: L400 + App Form

- Exception: Mismatch → Decision Impact: Refer Ops
- QnB=YES, EzSub=YES, Hardcopy=YES

Step 4C — Client patching (merge/void duplicates)

### Category 5: Client Information Checks

Step 5A — Check if client name sequence on L400 matches MyInfo
- Input: Name fields
- Source: L400 + MyInfo
- AI Role: Evaluate
- Rule IF: Match → THEN: Pass
- Exception: Mismatch → Decision Impact: Refer Ops
- QnB=NO, EzSub=NO, Hardcopy=NO

Step 5B — Update Race from OTH to correct value on L400
- Input: Race
- Source: L400
- AI Role: Evaluate
- Rule IF: OTH and Doc shows CHI → THEN: Recommend update
- Exception: None → Decision Impact: None
- QnB=NO, EzSub=NO, Hardcopy=NO

Step 5C — Check key fields in L400 vs Application Form
- Input: Sex, Address, SMS, Email, Nationality, Identity No, Birth Date
- Source: L400 + App Form
- AI Role: Evaluate
- Rule IF: All match → THEN: Pass
- Rule IF: Any mismatch → THEN: Fail
- Exception: Mismatch → Decision Impact: Further Requirements
- Why each field matters:
  * Sex: Gender affects premium
  * Address: PDPA compliance (prevents return mail)
  * SMS: PDPA compliance (mobile number)
  * Email: Client needs for MySinglife account registration
  * Nationality: Compliance related
  * Identity No: Compliance related
  * Birth Date: ANB affects premium
- IMPORTANT: Information may be wrong due to concurrent applications (e.g. EzSub submitted after, fields keyed wrongly due to manual entry)
- QnB=YES, EzSub=YES, Hardcopy=YES

Step 5D — Update incorrect fields on L400
- Input: Client fields
- Source: L400
- AI Role: Recommend (human-only action)
- Exception: Human-only → Decision Impact: Refer Ops
- QnB=NO, EzSub=NO, Hardcopy=NO

Step 8E — Check Postpone value

## DECISION MAPPING

| Condition | AI Recommendation |
|-----------|------------------|
| All applicable steps Pass; no outstanding PF | Standard |
| Any doc missing OR PF not all R OR unresolved data mismatch | Standard with Further Requirements |
| Any UW indicator triggers OR RCS result = Refer | Refer to UW |
| C09 outstanding or compliance hit | Trigger GNS / Compliance |
| Withdrawal request received | Withdrawal |

## WORKFLOW ACTION LABELS

| Action | Human Review Required |
|--------|----------------------|
| Decision Locked with No Further Requirements | No |
| Trigger follow-up / pending requirements | No |
| Underwriting Review | Yes |
| Compliance workflow | Yes |
| Close case (withdrawal) | No |

## DECISION OUTCOMES (Detailed Steps)

### A) Standard — Accept with No Further Requirements
When to use: All steps pass, no outstanding PF, no UW triggers.
Steps:
1. Go to 'Underwriting Decision Update' screen on L400
2. Set 'Proposal Accepted' = Y
3. Set 'Print Acceptance Letter' = B
4. Go to DotSphere
5. Select Decision: 'Decision Locked with No Further Requirements'

### B) Standard with Further Requirements
When to use: No UW trigger, but outstanding items exist (missing docs, PF not all R, data mismatch).
Steps:
1. Go to 'Proposal Follow-ups' screen on L400
2. Set 'Print Letter' = B
3. Go to DotSphere
4. Select Decision: 'All Data Verified?' + 'Mandatory PF/BI/FFF Missing Info?' + 'Initial/Continue Pending Requirement with Original NTU days'

### C) Refer to UW
When to use: Any Jet UW indicator triggered (Y/M) or SA exceeds threshold or RCS = Refer.
Steps:
1. Go to DotSphere
2. Select Decision: 'Underwriting Review?'

### D) Trigger GNS Review
When to use: C09 follow up code is 'O' OR fields were amended requiring GNS check.
Steps:
1. Go to DotSphere
2. Select Decision: 'Trigger GNS review'

### E) Trigger Compliance
When to use: GNS follow up code is 'O' OR there is a GNS hit.
Steps:
1. Go to DotSphere
2. Select Decision: 'Compliance Docs Received/Trigger Compliance'

### F) Withdrawal Request Received
When to use: Withdrawal request is received.
Steps:
1. Go to 'Contract Issue' screen on L400
2. Enter Contract Number
3. Action = B
4. Reason = N001
5. Go to DotSphere
6. Select Decision: 'Withdrawal Request Received'

## EXCEPTIONS / KNOWLEDGE GAPS

Flag and log for clarification if any of the following occurs:
- Ambiguous or missing scenario rules
- Missing definitions of specific follow-up codes
- Unclear thresholds or reference lists (e.g., ICD pass/fail list for RCS checks)
- Multiple client records found with no clear resolution path
- Any rule/threshold that is missing or unclear — do NOT guess
