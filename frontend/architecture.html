End-to-End Architecture

1. Input Layer

SharePoint docs, case data (JSON), user queries. Sources: L400 extractions, FileNet metadata, application forms.

→

2. Privacy Filter

Masks NRIC, names, phone, email, addresses before LLM. Rules engine sees raw data; LLM sees sanitized data only.

→

3. Rules Engine

Deterministic SOP evaluation from sop_rules.json. 15 steps, channel gating, threshold checks. No LLM involved.

↓

6. Output

Evaluation report, email draft, QA review, or Q&A answer. PII restored for ops staff view. All outputs are drafts for human review.

←

5. LLM (Claude)

Explains decisions, drafts emails, answers questions. Uses ONLY RAG context. Never sees raw PII. Never makes decisions.

←

4. RAG Retrieval

ChromaDB vector store, 89 chunks from 21 knowledge base files. Semantic search returns top-k relevant SOP sections.

LLM Role Boundary

  • LLM DOES: Explain decisions, draft emails, answer SOP queries, summarize QA reviews
  • LLM DOES NOT: Make decisions, modify data, auto-send emails, override rules engine
  • Rules Engine DOES: All deterministic checks — field matching, threshold evaluation, follow-up status, UW indicators
  • Privacy Filter DOES: Mask PII before LLM, restore PII in output for ops staff

Use Cases

UC1: SOP Evaluation (QnB/EzSub CareShield)

Rules engine evaluates case → Claude explains → Ops acts

  • Input: Case JSON (client fields, follow-ups, UW indicators)
  • Processing: 15-step deterministic rule evaluation
  • Output: Step-by-step report + decision + DotSphere actions

UC2: Customer Email Drafting

UW decides → AI drafts empathetic email → Human reviews

  • Input: Decision type, tone, customer context
  • Processing: Email rules + compliance guardrails
  • Output: Analysis + compliant email draft (no medical disclosure)

UC3: QA Underwriting Review

QA indicators → Complexity scoring → Risk drivers explained

  • Input: ICD codes, loadings, exclusions, turnaround time
  • Processing: 7-indicator scoring (deterministic) + Claude explanation
  • Output: Risk level, score breakdown, QA recommendation

UC4: SOP Knowledge Q&A

Staff asks → RAG retrieves → Claude answers from SOPs only

  • Input: Natural language question
  • Processing: Semantic search over 21 KB documents (89 chunks)
  • Output: SOP-grounded answer with references

Technology Stack

Backend

  • Python / Flask — API server
  • Rules Engine — deterministic SOP checks from JSON config
  • ChromaDB — vector store for RAG
  • sentence-transformers — local embeddings (no API cost)
  • Claude API (Sonnet) — explanation & email drafting

Frontend

  • Vanilla HTML/CSS/JS — no framework dependency
  • SSE streaming — real-time response display
  • 4 modes: Chat, Evaluate, Draft Email, QA Review
  • LocalStorage — conversation persistence
